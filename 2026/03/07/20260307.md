    Basic Usage
    ===========
    
    ``dill`` is a drop-in replacement for ``pickle``. Existing code can be
    updated to allow complete pickling using::
    
        >>> import dill as pickle
    
    or::
    
        >>> from dill import dumps, loads
    
    ``dumps`` converts the object to a unique byte string, and ``loads`` performs
    the inverse operation::
    
        >>> squared = lambda x: x**2
        >>> loads(dumps(squared))(3)
        9

    The default serialization can also be set globally in *dill.settings*.
    Thus, we can modify how ``dill`` handles references to the global dictionary
    locally or globally::
    
        >>> import dill.settings
        >>> dumps(absolute) == dumps(absolute, recurse=True)
        False
        >>> dill.settings['recurse'] = True
        >>> dumps(absolute) == dumps(absolute, recurse=True)
        True
    
    ``dill`` also includes source code inspection, as an alternate to pickling::
    
        >>> import dill.source
        >>> print(dill.source.getsource(squared))
        squared = lambda x:x**2

    Pickle an object to a file.

dumps(obj, protocol=None, byref=None, fmode=None, recurse=None, **kwds)
    Pickle an object to a string.
    
    *protocol* is the pickler protocol, as defined for Python *pickle*.
    
    If *byref=True*, then dill behaves a lot more like pickle as certain
    objects (like modules) are pickled by reference as opposed to attempting
    to pickle the object itself.
    
    If *recurse=True*, then objects referred to in the global dictionary
    are recursively traced and pickled, instead of the default behavior
    of attempting to store the entire global dictionary. This is needed for
    functions defined via *exec()*.
    
    *fmode* (:const:`HANDLE_FMODE`, :const:`CONTENTS_FMODE`,
    or :const:`FILE_FMODE`) indicates how file handles will be pickled.
    For example, when pickling a data file handle for transfer to a remote
    compute service, *FILE_FMODE* will include the file contents in the
    pickle and cursor position so that a remote method can operate
    transparently on an object with an open file handle.
    
    Default values for keyword arguments can be set in :mod:`dill.settings`.

import os
import sys
from functools import wraps

def dump_runtime_environment(func):
    @wraps(func)
    def decorator(*args, **kwargs):
        if not os.getenv("DEBUG"):
            return func(*args, **kwargs)

        import dill

        captured = {}

        def profiler(frame, event, arg):
            if event == "return" and frame.f_code is func.__code__:
                captured["locals"] = frame.f_locals.copy()
                captured["globals"] = frame.f_globals
            return profiler

        old_profiler = sys.getprofile()
        sys.setprofile(profiler)
        try:
            result = func(*args, **kwargs)
        finally:
            sys.setprofile(old_profiler)

        run_info = {
            "args": args,
            "kwargs": kwargs,
            "result": result,
            "func": captured,  # locals/globals right before return
        }

        with open(f"{func.__name__}.dill", "wb") as f:
            dill.dump(run_info, f, byref=True, recurse=True)

        return result

    return decorator



@dump_runtime_environment
def main():
    cell = ('.', 'C', '/', '\\')
    direction = ('U', 'R', 'D', 'L')
    
    N, M = [int(char) for char in input().split()]
    for _ in range(N):
        print("===")
        for c in input():
            print(c)


if __name__ == "__main__":
    main()

Perfect — you’re essentially looking to **generalize your `dump_runtime_environment` system** to real-world Python workflows. The common theme is **wrapping a function or entry point in a decorator** that captures locals/globals and optionally triggers `ipdb` or saves `.dill`. I’ll go through each case with professional patterns.

---

## 1️⃣ **FastAPI**

FastAPI apps are event-driven: your “runtime function” is usually a path operation function.

**Example:**

```python
from fastapi import FastAPI
from utils import dump_runtime_environment

app = FastAPI()

@dump_runtime_environment
@app.get("/add")
def add(x: int, y: int):
    result = x + y
    return {"result": result}
```

**Notes:**

* The decorator works just like before.
* If `DEBUG` is enabled, each request will **dump args, kwargs, locals, globals** before returning.
* You can also use **middleware** to capture **request/response**:

```python
from starlette.middleware.base import BaseHTTPMiddleware

class DumpMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        response = await call_next(request)
        if os.getenv("DEBUG"):
            import dill
            dill.dump({"path": request.url.path}, "fastapi_request.dill")
        return response

app.add_middleware(DumpMiddleware)
```

This is handy for **capturing full request context**.

---

## 2️⃣ **Flask (registering as a command)**

Flask 2+ supports **custom CLI commands** via `@app.cli.command`.

```python
from flask import Flask
from utils import dump_runtime_environment

app = Flask(__name__)

@app.cli.command("run-task")
@dump_runtime_environment
def run_task():
    print("Running some Flask task")
    x = sum(range(10))
    print(x)
```

**Usage:**

```bash
export DEBUG=1
flask run-task
```

* `dump_runtime_environment` wraps the task function.
* `.dill` is generated automatically.
* Works for both web request handlers or custom CLI commands.

---

## 3️⃣ **Airflow DAG tasks**

Airflow tasks are just Python callables wrapped with Operators (PythonOperator, etc.).

```python
from airflow.decorators import dag, task
from datetime import datetime
from utils import dump_runtime_environment

@dag(start_date=datetime(2026, 3, 1), schedule_interval=None)
def my_dag():

    @task
    @dump_runtime_environment
    def process_data():
        data = [1,2,3]
        total = sum(data)
        print(total)
        return total

    process_data()

dag = my_dag()
```

**Key points:**

* Wrap the **Python callable inside `@task`** with your decorator.
* `.dill` will capture **locals/globals of the task execution**.
* Works even when Airflow executes tasks in separate processes.

---

## 4️⃣ **Superset custom CLI**

Superset uses **Click commands**. You can wrap your command function:

```python
from superset import app
from flask.cli import with_appcontext
from utils import dump_runtime_environment

@app.cli.command("my-custom-task")
@with_appcontext
@dump_runtime_environment
def my_custom_task():
    print("Running Superset CLI task")
    value = 42
    print(value)
```

**Run:**

```bash
export DEBUG=1
superset my-custom-task
```

* `with_appcontext` ensures Flask/Superset app context is loaded.
* Your decorator captures runtime environment just like in `main.py`.

---

## ✅ **General patterns**

Across all these frameworks:

1. **Decorator first:** `@dump_runtime_environment` should wrap the function you care about.
2. **Optionally set DEBUG:** `os.environ["DEBUG"] = "1"` triggers dumping.
3. **Handle exceptions:** Let your decorator capture exception + locals.
4. **Framework integration points:**

| Framework | Entry point / function                              |
| --------- | --------------------------------------------------- |
| FastAPI   | Path operation function / Middleware                |
| Flask     | CLI commands (`@app.cli.command`) or route handlers |
| Airflow   | PythonOperator callable / `@task`                   |
| Superset  | Click CLI commands / `@with_appcontext`             |

---

## 5️⃣ **Optional: unified dumper for frameworks**

You could make a helper:

```python
def framework_dump(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        import os, dill, sys
        try:
            return func(*args, **kwargs)
        except Exception as e:
            if os.getenv("DEBUG"):
                dill.dump({"args": args, "kwargs": kwargs, "exception": e}, f"{func.__name__}.dill")
                import ipdb; ipdb.post_mortem()
            raise
    return wrapper
```

Then apply it to **any callable**: FastAPI, Flask CLI, Airflow task, Superset CLI.
